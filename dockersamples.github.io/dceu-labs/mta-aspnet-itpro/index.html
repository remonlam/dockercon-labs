Modernizing ASP.NET Applications - for IT Pros

Oct 12, 2017 • @sixeyed

Moving your app to Docker is the easiest way to add portability, security and efficiency - without changing code. This hands-on lab shows you how to take an existing ASP.NET WebForms app in an MSI package, run it in Docker and use the platform for automated updates.

If you aren’t familar with the basic concepts of Docker, you can start with the Docker Windows 101 lab before running this lab.

Task 1: Packaging the MSI to run in a container

You’re going to start with an MSI that deploys a web app onto a server, which needs IIS and ASP.NET to be configured to run the app. You’ll package that MSI to run in Docker, using an image from Microsoft with all the dependencies already installed.

If you already have a scripted build process then you may have MSIs or Web Deploy packages already being generated, and it’s easy to package them into a Docker image.

Start by cloning the Git repository for the sample app:

git clone https://github.com/dockersamples/hol-mta-aspnet-itpro

cd hol-mta-aspnet-itpro

And now build version 1.0 of the application:

docker build --tag dockersamples/modernize-aspnet-itpro:1.0 .\v1.0

While the image is building, have a look at the Dockerfile for the Web application. It’s very simple:

FROM microsoft/aspnet:windowsservercore-10.0.14393.1715
COPY UpgradeSample-1.1.0.0.msi /
RUN Start-Process msiexec.exe `
     -ArgumentList '/i', 'C:\UpgradeSample-1.0.0.0.msi', 'RELEASENAME=2017.10', `
     '/quiet', '/norestart' -NoNewWindow -Wait

There are just three lines of Dockerfile instructions:

  • FROM specifies the base image to use as a starting point, in this case a specific version of the ASP.NET image
  • COPY copies the existing v1.0 application MSI from the local machine into the Docker image
  • RUN installs the MSI using msiexec, and passing a value to the custom RELEASENAME variable that the MSI uses. The install is wrapped in a PowerShell process with the -Wait flag, which makes sure it completes

Using MSIs is a quick way to get started with Docker, but your Dockerfile is not a clear deployment document. All the installation logic is in the MSI. An alternative is to compile the application from source in a multi-stage Dockerfile, like this example.

The sample application for the lab is a simple ASP.NET WebForms app, which the MSI installs to the default IIS website running on port 80. To start the application, use docker container run and publish the port from the container, so the website is available externally:

docker container run -d -p 80:80 --name v1.0 dockersamples/modernize-aspnet-itpro:1.0

When the run command completes, your application is running in Docker. You can list all the running containers:

docker container ls

The HTTP port for the web container is published so the app is available to the outside world.

Browse to version 1.0 of the app

The content is served by the container, and you’ll see a very simple demo app which just lists some basic environment details:

You can get basic management information about the container from Docker. Use the top command to see all the process running inside the container:

docker container top v1.0

You can see that w3wp.exe is running, which is the IIS worker process serving requests. Use the logs command to see the output from the container application logs:

docker container logs v1.0

In this case, IIS doesn’t write any logs to the console so there won’t be any log entries surfaced to Docker. In the next task, you’ll fix that so the IIS logs get relayed to Docker.

Task 2: Deploying application updates and Windows updates

In this task you deploy a new version of the application. To do that you’ll build a new image, and then replace the existing container with a new container.

The new version upgrades the version of the app and includes a Windows update, and it also integrates the application logs with Docker. Still without touching the code.

Build version 1.1 of the application:

docker image build --tag dockersamples/modernize-aspnet-itpro:1.1 .\v1.1

While the image is building, have a look at the Dockerfile for version 1.1. It takes the same approach as version 1.0, but it’s based on a later version of the Windows Server Core image, and it deploys a later version of the application MSI.

There are also two new parts in the Dockerfile. This RUN instruction configures how IIS writes log entries:

RUN Set-WebConfigurationProperty -pspath 'MACHINE/WEBROOT/APPHOST' -filter 'system.applicationHost/log' -name 'centralLogFileMode' -value 'CentralW3C'; `
    Set-WebConfigurationProperty -pspath 'MACHINE/WEBROOT/APPHOST' -filter 'system.applicationHost/log/centralW3CLogFile' -name 'truncateSize' -value 4294967295; `
    Set-WebConfigurationProperty -pspath 'MACHINE/WEBROOT/APPHOST' -filter 'system.applicationHost/log/centralW3CLogFile' -name 'period' -value 'MaxSize'; `
    Set-WebConfigurationProperty -pspath 'MACHINE/WEBROOT/APPHOST' -filter 'system.applicationHost/log/centralW3CLogFile' -name 'directory' -value 'c:\iislog'

There’s a lot there, but it’s just using PowerShell to configure IIS. It sets up IIS to write a single log file in a known location. Then the CMD instruction changes how Docker starts a container from the image:

CMD Start-Service W3SVC; `
    Invoke-WebRequest http://localhost -UseBasicParsing | Out-Null; `
    netsh http flush logbuffer | Out-Null; `
    Get-Content -path 'c:\iislog\W3SVC\u_extend1.log' -Tail 1 -Wait

This PowerShell scripts runs four actions when a container starts:

  • start the IIS Windows Service
  • make a call to the local website inside the container, which causes IIS to write a log entry
  • flush the IIS log buffer so the log file is written to disk
  • tail the log file, so when IIS writes log entries they get relayed to Docker.

When the build finishes, you have two application images, tagged 1.0 and 1.1. Each contains different versions of the application built on different versions of Windows, with a different startup configuration. You can see the basic image details with the docker image ls command:

docker image ls --filter reference='*/modernize*'

You’ll see output like this:

REPOSITORY                            TAG   IMAGE ID      CREATED          SIZE
dockersamples/modernize-aspnet-itpro  1.1   05781f223267  2 minutes ago    11.9GB
dockersamples/modernize-aspnet-itpro  1.0   d24e10dfe126  20 minutes ago   11.6GB

You can see the images are listed at over 11GB each, but this is the logical size, and the images actually share a common set of data files - the physical storage they use is much smaller.

You can check that by listing all images:

docker image ls

And then checking the storage Docker is using:

docker system df

You’ll see about 20 images that are pre-loaded in the lab environment. All the logical sizes added together is around 150GB. But because Docker shares image files, those 20 images take up around 30GB of disk space.

Run the new version of the application to see what’s changed:

docker container run -d -p 8080:80 --name v1.1 dockersamples/modernize-aspnet-itpro:1.1

Browse to version 1.1 of the app

The new version looks like this:

Refresh the browser several times, and then check back to see the container logs:

docker container logs v1.1

You’ll see the standard IIS access logs, relayed by the container into the Docker platform. The new container is running the new application, but the old app is still running in the original container. The Docker platform is very efficient, and you can run many containers on a single server. In the last task you’ll learn how to do rolling updates and rollbacks.

Task 3: Rolling out and rolling back updates

The Docker platform supports automated rolling updates, so you can have a zero-downtime update of your app. To get zero-downtime deployment you need to run your containers as a service in a Docker swarm - a cluster of machines all running Docker, which you manage as a single unit.

To update a service Docker incrementally stops the running containers and starts new ones from the updated image. For production environments where you have a load-balancer directing traffic to nodes, it would not send any requests to a node which Docker is updating, so your application stays online.

Start by cleaning up all the running containers:

docker container rm -f $(docker container ls -a -q)

Now switch to swarm mode. Run the swarm init command to initialize the swarm:

docker swarm init --advertise-addr 127.0.0.1

This is a single-node swarm. You don’t get high availability with a single node, but to join multiple servers to the swarm you just need to install Docker on each and run the docker swarm join command you see in the output.

This node is now a manager, a server which controls the swarm.

Run version 1.0 of the web app on your swarm, creating it as a service:

docker service create `
  --name sample `
  --publish mode=host,target=80,published=80 `
  --detach=true `
  --replicas=1 `
  dockersamples/modernize-aspnet-itpro:1.0
  • --name - is the name of the service, this is how you refer to the service for management, and in other services that consume this one
  • --publish - maps the container port to the host port. Note that the syntax is different in swarm mode, but the functionality is the same
  • --replicas - the number of containers that run the service. Docker will maintain the service level by starting new containers when needed

Check on the service with docker service ls:

docker service ls

With a single-node swarm and a single replica, you’ll see one container on the manager node.

Browse to the app - this is version 1.0

It’s the same application running in a container, but now you can manage it as a swarm service, which provides update and rollback.

Update your app to version 1.1:

docker service update --image dockersamples/modernize-aspnet-itpro:1.1 sample

That tells Docker to update the sample service to version 1.1 of the image. With a multi-node swarm and many service replicas, Docker updates one container at a time. In production you would configure a load balancer in front the nodes, so no traffic gets sent to nodes which are being update and you get zero-downtime deployment.

Check on the services again. and you will see there is still one replica, but it is running version 1.1 of the app:

docker service ls

Browse to the app - this is the same URL but now the app is version 1.1

Automated updates are a huge benefit of the Docker platform. Updating a distributed application in a safe, automated way takes all the risk out of deployments, and makes frequent releases possible. If you don’t like the new color scheme in v1.1 you can easily roll back to v1.0:

docker service update --rollback sample

Check on the services again. and you will see there is still one replica, but it is running version 1.0 of the app:

docker service ls

Browse to the app - this is the same URL but now the app has reverted to version 1.0

Rolling back is the same as updating. The existing containers are stopped, and new containers created using the original image version. Refresh the app site and you will see version 1.0 running again. Automated rollback may be an even bigger benefit than automatic update. Knowing you can revert to the previous good version without any downtime and without a lengthy manual procedure gives you confidence in your deployment process, even if there are problems in the application itself.

Conclusion

You now know how to package an existing ASP.NET app as a Docker image, without touching the source code. You also know how to package a new version of your image, containing application updates and the latest Windows patches. Lastly you’ve seen how to run a highly-available service in Docker swarm mode.
